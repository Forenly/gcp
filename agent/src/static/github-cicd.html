The Pull Request workflow

At Forenly, every change reaches users the same way: it goes onto a branch, into a Pull Request the team can review, gets merged to main, and is deployed from main. This guide walks that path end to end, with real screenshots.

The principle

A running demo is not “done.” Work is done when it’s merged to main and deployed from main — reviewable, reproducible, and owned by the team.

Code always starts on someone’s machine. A live URL only proves it ran once, there — the team still can’t read it, diff it, comment on it, or reproduce it, and a local build can quietly drift behind main. The path below is how any change gets from a laptop into the project the same way, every time.

The goal: your change becomes a Pull Request people can open and read, gets merged into main, and the live site is then deployed from main — not from a laptop.

OverviewThe flow at a glance

Four words do all the work. Think of them as directions — what moves where:

pulldown · GitHub → you

Get everyone’s latest as your starting point. Does not publish your work.

commitsave · local

Record your edits onto your branch as a checkpoint.

pushup · you → GitHub

Upload your branch so the team can see it.

open a PRpropose

Ask for your branch to go into main. Review happens here.

The full path, shown with a real Lawn Advisor fix — the map’s Draw lawn button was throwing getPath() of undefined:

  1. pull — git checkout main && git pullStart from the latest main, before branching.
  2. branch — git checkout -b fix/draw-lawn-getpathYour own line off main, named for the fix.
  3. editMake the actual change — the Draw-lawn handler now passes paths:[[]].
  4. commit — git commit -m "fix(map): Draw lawn threw getPath() of undefined"Save the change locally as a checkpoint.
  5. push — git push -u origin fix/draw-lawn-getpathUpload the branch to GitHub — now the team can see it.
  6. open a PRbase main ← compare fix/draw-lawn-getpath.
  7. review / approveOn the PR, after push — a teammate reads the diff.
  8. fix if askedMore commits → push again; they appear in the PR automatically.
  9. mergeThe PR goes into main.
  10. deployDeploy the live site from main.

The rest of this guide is these same steps, hands-on, with screenshots.

Part A · On your computerGet the code out of your machine

Everything here is in the terminal, inside the project folder. This is where work leaves a laptop and reaches GitHub.

How branches work on a team

No one commits directly to main — it’s the shared, reviewed trunk that everyone deploys from. Every change starts as its own branch off the latest main. It’s one branch per change (a feature or a fix), not one permanent branch per person: you branch, work, open a PR, it merges into main, you delete the branch — and the next change starts a fresh branch again.

1

See what you changed

git status

This lists the files you touched. If your work shows up here, it exists locally — but nowhere else yet.

2

Put your work on a branch

git checkout -b fix/my-change

A branch keeps your work separate from main until it’s reviewed. Name it for what it does.

3

Commit it

git add -A
git commit -m "describe what changed and why"
4

Push the branch to GitHub

git push -u origin fix/my-change

This is the key step. After it, the code is on GitHub — visible to the team — instead of only on one machine.

Part B · On GitHubOpen a Pull Request

The Pull Request (PR) is the unit of review — where the team reads a change and discusses it. The diff is the conversation.

5

Go to the Pull requests tab

6

Choose the branch and create it

7

Title it, describe it, submit

Add a short title and a sentence on what and why. Then create the PR. Reviewers comment; you push more commits to the same branch and they appear in the PR automatically. When it’s approved, it gets merged into main.

Part C · After mergeDeploy from main, not a laptop

8

Redeploy from the merged code

Once the change is in main, the live site is deployed from main — so what users see equals what the team reviewed.

git checkout main
git pull
# then run the project's deploy from this clean main checkout
Already shared a local build or URL?

Common, and easy to close the loop: open a PR with that exact code (Parts A–B), get it merged, then redeploy from main. Often the live version is simply an older build, and the fix is a deploy, not new code. Then re-share: “Deployed from main @ <commit> — PR #N.”

Checklist before you say “done”

  • The change is on a branch, committed, and pushed (the branch is visible on GitHub).
  • There is a PR others can open and read.
  • What’s deployed was built from main, not a local copy.
  • The branch was up to date with main (no silent drift / missing fixes).
  • The link being shared points at code that exists in the repo.
